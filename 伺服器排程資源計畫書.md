# 業務日誌 LLM 萃取系統 — 伺服器排程資源計畫書

**版本**：v1.0  
**建立日期**：2026-05-04  
**適用場景**：日報新增後自動化流程執行

---

## 1. 目標與範圍

### 1.1 核心目標
- **增量處理**：每日自動偵測新增的業務日報（CRMGY 表新資料），自動執行完整 LLM 萃取 pipeline
- **端到端自動化**：從原始日報 → Phase L → Phase 0 → Phase M → Phase 1 → Phase 2 → Phase 3，全程無人工介入
- **資源最佳化**：僅處理增量資料，避免重複計算歷史資料
- **高可用性**：提供錯誤重試、進度追蹤、失敗告警機制

### 1.2 適用場景
- **日常維護**：每日早上 6:00 執行，處理前一日新增的業務日報
- **週末補跑**：週一早上 7:00 執行，處理週末累積的日報
- **手動觸發**：透過 API 或命令列手動執行特定日期範圍

### 2.1 完整流程圖

```
┌─────────────────────────────────────────────────────────────┐
│ 觸發條件：cron job (每日 06:00) 或手動 API 呼叫            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase L：種子庫維護 (每週日執行，非每日)                    │

│ - 重新索引 Vertex AI Vector Search                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 0：增量日報擷取                                       │
│ - SQL 查詢：WHERE CREATE_DATE >= @last_run_date            │

│ ┌─────────────────────────┐ ┌─────────────────────────┐    │

│ └─────────────────────────┘ └─────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 1：L1-L3 深度標籤 (串聯執行)                          │
│ - L1 痛點類型 KNN (k=5, LD001-LD005)                        │
│ - L2 客戶類型 KNN (k=5, LD006-LD010)                        │
│ - L3 解決方案 KNN (k=5, LD011-LD015)                        │

└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 2：L4-L7 三層標籤 (串聯執行)                          │

│ - 分層 3：KNN 相似樣本投票 (k=30)                            │

                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 3：合約與客戶行為分析 (串聯執行)                      │
│ - 3.1 合約資料擷取 (SQL JOIN: CRMGY + 合約表)               │
│ - 3.2 客戶分類 (續約客/即將到期/升級客/新客/流失客/沉睡潛客) │
│ - 3.3 活動內容解析 (Gemini Flash: PDF/Word/圖片 → 摘要+標籤)│
│ - 3.4 沉睡客戶話術生成 (Gemini Flash: 150-200 字 + 活動連結)│
│ - 3.5 續約路徑分析 (產品升級追蹤、交叉銷售關聯規則)         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 輸出層：結果匯總與儲存                                      │
│ - CSV 輸出：完整標籤結果 (Phase M/1/2/3 合併)               │
│ - SQL 寫回：更新 CRMGY 或專屬標籤表                         │
│ - Log 記錄：執行時間、成功/失敗筆數、錯誤訊息               │

### 2.2 關鍵依賴關係
- **Phase M 與 Phase 1 平行**：商機等級/法人標籤可與 L1-L3 同時執行（無依賴）
- **Phase 2 依賴 Phase M + Phase 1**：需要 Stage 與 L1 Top-1 標籤進行分層
- **Phase 3 依賴 Phase M**：需要 Stage 欄位進行客戶分類
- **Phase 3.4 依賴 Phase 1 + Phase 2**：話術生成需要完整 L1-L7 標籤

**SQL 查詢最佳化**：

-- 增量查詢（使用索引）
SELECT 
    GY001, GY003, GY004, GY022, GY012, CREATE_DATE
FROM CRMGY WITH (NOLOCK)
WHERE CREATE_DATE >= @last_run_date
  AND GY012 IS NOT NULL
  AND LEN(LTRIM(RTRIM(GY012))) > 5
ORDER BY CREATE_DATE DESC

-- 建議索引
CREATE INDEX idx_crmgy_create_date ON CRMGY(CREATE_DATE) INCLUDE (GY001, GY012)
```

### 4.3 錯誤處理與重試機制

#### Gemini API 呼叫重試
```python
import time
from functools import wraps

def retry_on_quota_error(max_retries=5, backoff_factor=2):
    """裝飾器：處理 Gemini API quota 錯誤"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if "quota" in str(e).lower() or "rate limit" in str(e).lower():
                        wait_time = backoff_factor ** attempt
                        logger.warning(f"API quota 超限，等待 {wait_time} 秒後重試（第 {attempt+1}/{max_retries} 次）")
                        time.sleep(wait_time)
                    else:
                        raise
            raise Exception(f"超過最大重試次數 {max_retries}")
        return wrapper
    return decorator

@retry_on_quota_error(max_retries=5)
def call_gemini_api(prompt):
    response = gemini_model.generate_content(prompt)
    return response.text
```

from sqlalchemy.pool import QueuePool

# 建立連線池（避免頻繁建立連線）
engine = create_engine(
    "mssql+pyodbc://...",
    poolclass=QueuePool,
    pool_size=5,          # 最多 5 個連線

def fetch_with_retry(query, max_retries=3):

            return pd.read_sql(query, engine)
        except Exception as e:
            logger.warning(f"SQL 查詢失敗（第 {attempt+1}/{max_retries} 次）：{str(e)}")
            if attempt < max_retries - 1:
                time.sleep(5)
    raise Exception("SQL 查詢失敗，超過最大重試次數")

## 9. 風險與應變計畫

### 9.1 高風險項目

| 風險項目                     | 可能影響                          | 應變措施                          |
|------------------------------|-----------------------------------|-----------------------------------|
| Gemini API Quota 超限        | Phase M/3 執行中斷                | 1. 自動重試（exponential backoff）<br>2. 申請提高 quota 至 120 RPM |
| Vertex AI 索引損毀           | Phase 1/2 KNN 查詢失敗            | 1. 保留上週種子庫備份<br>2. 自動重建索引（Phase L 重跑） |
| SQL Server 連線逾時          | Phase 0/3 資料擷取失敗            | 1. 連線池自動重連<br>2. 拆分大批次查詢（OFFSET/FETCH） |
| 記憶體不足（OOM）            | Phase 2 歷史樣本載入時崩潰        | 1. 升級至 64GB RAM<br>2. 改用資料庫分頁查詢（減少記憶體佔用） |
| 磁碟空間滿                   | 無法寫入輸出 CSV                  | 1. 自動清理 30 天前備份<br>2. 壓縮歷史 CSV |

### 9.2 災難復原

- **備份頻率**：
  - SQL Server：每日自動備份（DBA 管理）
  - Vertex AI 索引：每週日 Phase L 執行後自動備份至 GCS
  - 輸出 CSV：每月壓縮備份至 NAS

- **復原時間目標（RTO）**：
  - SQL 連線中斷：< 5 分鐘（自動重連）
  - 種子庫索引損毀：< 4 小時（重跑 Phase L）
  - 伺服器故障：< 1 天（重新部署至備援 VM）
