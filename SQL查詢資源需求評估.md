# SQL 查詢資源需求評估

**評估日期**：2026-05-05  
**評估範圍**：SQL Server 資料查詢對現有系統的影響

---

## 1. 查詢負載概覽

### 1.1 查詢類型與頻率

| 查詢類型                 | 執行頻率       | 涉及表格              | 預估資料量        |
|--------------------------|----------------|-----------------------|-------------------|
| Phase 0：增量日報查詢    | 每日 1 次      | CRMGY                 | 685 筆/天         |
| Phase 3：合約資料 JOIN   | 每日 1 次      | CRMGY + 合約表        | 685 × 3 筆關聯    |
| Phase 3：續約分析 GROUP BY | 每日 1 次    | 合約表                | 2,055 筆聚合      |

### 1.2 執行時段

- **啟動時間**：每日 06:00（伺服器排程）
- **執行時長**：日均 ~3-5 分鐘（SQL 查詢部分）
- **峰值時長**：~8-12 分鐘（1,030 筆情境）
- **低峰時段**：避開上班時段（09:00-18:00），對業務零影響

## 2. 資料傳輸量評估

### 2.1 單日傳輸量（三種情境）

| 情境         | Phase 0 日報查詢 | Phase 3 合約 JOIN | 總計      |
|--------------|------------------|-------------------|-----------|
| 日均（685 筆）| 1.3 MB          | 3.1 MB            | **4.4 MB**|
| 峰值（1,030 筆）| 2.0 MB        | 4.6 MB            | **6.6 MB**|
| 極端（1,356 筆）| 2.7 MB        | 6.1 MB            | **8.8 MB**|

**評估結論**：
- ✅ 單日資料量極小（< 10 MB），對網路頻寬影響微乎其微
- ✅ 年累積傳輸量：4.4 MB × 365 = **1.6 GB/年**（可忽略）

### 2.2 網路頻寬需求

**峰值頻寬**：
- 資料傳輸時間：~5 分鐘（300 秒）
- 峰值頻寬：8.8 MB / 300 秒 = **0.29 Mbps**
- 相較企業內網（通常 1 Gbps）：佔用率 < 0.03%

**評估結論**：
- ✅ 網路頻寬需求極低，不會造成網路瓶頸
- ✅ 即使同時有其他系統查詢，影響仍可忽略

## 3. SQL Server 資源消耗

### 3.1 CPU 使用率

| 查詢類型                 | 無索引        | 有索引（建議）| 說明                          |
|--------------------------|---------------|---------------|-------------------------------|
| Phase 0：增量查詢（WHERE）| 8-12%        | **2-5%**      | 日期範圍篩選 + 長度檢查       |
| Phase 3：JOIN 查詢        | 15-25%       | **5-8%**      | CRMGY × 合約表（1:3 關聯）    |
| Phase 3：GROUP BY 聚合    | 10-18%       | **3-6%**      | 續約路徑分析                  |

**平均 CPU 負載**（日均情境，有索引）：
- 查詢期間：~5-8%
- 平均每日影響：5% × (5分鐘 / 1440分鐘) = **0.017%**（可忽略）

**評估結論**：
- ✅ CPU 消耗極低，對 SQL Server 主機影響微小
- ✅ 建議建立索引後，CPU 使用率可降低 60-80%

---

### 3.2 記憶體佔用

| 查詢類型                 | 記憶體消耗    | 說明                          |
|--------------------------|---------------|-------------------------------|
| Phase 0：增量查詢        | 30-50 MB      | 查詢結果集（685 筆 × 6 欄位） |
| Phase 3：JOIN 查詢       | 80-120 MB     | 中間結果集（JOIN 緩衝）       |
| Phase 3：GROUP BY 聚合   | 50-80 MB      | 聚合運算緩衝區                |

**峰值記憶體**：~120 MB（Phase 3 JOIN 查詢）

**評估結論**：
- ✅ 記憶體消耗極小（< 150 MB）
- ✅ 相較 SQL Server 典型配置（64-256 GB），佔用率 < 0.1%
- ✅ 查詢完成後記憶體立即釋放，無累積效應

---

### 3.3 磁碟 I/O（IOPS）

| 查詢類型                 | 無索引 IOPS   | 有索引 IOPS   | 改善幅度      |
|--------------------------|---------------|---------------|---------------|
| Phase 0：增量查詢        | 1,500-2,000   | **200-500**   | ↓ 75%         |
| Phase 3：JOIN 查詢       | 5,000-8,000   | **800-1,200** | ↓ 85%         |
| Phase 3：GROUP BY 聚合   | 3,000-5,000   | **500-800**   | ↓ 83%         |

**平均 IOPS**（有索引）：~500-800 IOPS

**評估結論**：
- ✅ 索引建立後，IOPS 需求大幅降低 75-85%
- ✅ 相較企業級 SSD（10,000+ IOPS），佔用率 < 10%
- ⚠️ **強烈建議**：建立索引以避免全表掃描（Table Scan）

## 4. 查詢效能分析

### 4.1 查詢時間比較（有/無索引）

| 查詢類型                 | 無索引        | 有索引        | 改善幅度      | SLA 目標      |
|--------------------------|---------------|---------------|---------------|---------------|
| Phase 0：增量查詢        | 15 秒         | **3 秒**      | ↓ 80%         | < 5 秒        |
| Phase 3：合約 JOIN       | 60 秒         | **12 秒**     | ↓ 80%         | < 20 秒       |
| Phase 3：續約分析        | 90 秒         | **20 秒**     | ↓ 78%         | < 30 秒       |

**總查詢時間**（所有查詢串聯）：
- 無索引：165 秒（~2.8 分鐘）
- 有索引：**35 秒**（~0.6 分鐘）
- 改善：**↓ 79%**

**評估結論**：
- ✅ 索引建立後，查詢時間從 2.8 分鐘縮短至 0.6 分鐘
- ✅ 符合 SLA 目標（總執行時間 < 1 小時）
- ✅ 為後續 Pipeline 留下充足時間緩衝

---

### 4.2 索引建議

#### 建議索引 1：CRMGY 表

```sql
-- 索引名稱：idx_crmgy_create_date_include
-- 用途：Phase 0 增量查詢優化
CREATE INDEX idx_crmgy_create_date_include 
ON CRMGY(CREATE_DATE) 
INCLUDE (GY001, GY003, GY004, GY022, GY012)
WITH (FILLFACTOR = 90, PAD_INDEX = ON);

-- 預估大小：~500 MB（基於 100 萬筆歷史日報）
-- 建立時間：~3-5 分鐘
-- 維護成本：每日新增 685 筆，索引更新 < 1 秒
```

**效益**：
- 查詢時間：15 秒 → 3 秒（↓ 80%）
- CPU 使用率：10% → 3%（↓ 70%）
- IOPS：2,000 → 300（↓ 85%）

---

#### 建議索引 2：合約表

```sql
-- 索引名稱：idx_contract_company_dates
-- 用途：Phase 3 合約 JOIN 優化
CREATE INDEX idx_contract_company_dates 
ON 合約表(company_id, contract_start_date, contract_end_date)
INCLUDE (product_id, contract_amount)
WITH (FILLFACTOR = 90);

-- 預估大小：~800 MB（基於 50 萬筆合約）
-- 建立時間：~5-8 分鐘
-- 維護成本：每月新增 ~1,000 筆合約，索引更新 < 2 秒
```

**效益**：
- JOIN 查詢時間：60 秒 → 12 秒（↓ 80%）
- CPU 使用率：20% → 6%（↓ 70%）
- IOPS：6,000 → 1,000（↓ 83%）

## 5. 對現有系統的影響評估

### 5.1 影響評估矩陣

| 影響層面             | 影響程度       | 說明                          | 風險等級   |
|----------------------|----------------|-------------------------------|------------|
| SQL Server CPU       | **極低**       | 平均負載 < 0.02%              | ✅ 低      |
| SQL Server 記憶體    | **極低**       | 峰值 < 150 MB（< 0.1%）       | ✅ 低      |
| 磁碟 I/O             | **低**         | 有索引：500-800 IOPS（< 10%） | ✅ 低      |
| 網路頻寬             | **極低**       | 峰值 0.29 Mbps（< 0.03%）     | ✅ 低      |
| 業務系統查詢         | **無影響**     | 執行於低峰時段（06:00-07:00） | ✅ 低      |
| 資料庫鎖定           | **極低**       | WITH (NOLOCK) 讀取，不鎖表    | ✅ 低      |
| 索引維護成本         | **低**         | 每日新增 < 2 秒               | ✅ 低      |

**總體評估**：✅ **對現有系統影響極低，可放心部署**

**評估結論**：
- ✅ 即使與 5-10 個其他系統併發查詢，SQL Server 仍在正常負載範圍內
- ✅ 早上 06:00 為業務低峰時段，併發可能性極低
